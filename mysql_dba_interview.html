📋 Main Interview Questions
01
🟢 Slow Query / Index Usage
This query runs on a 50M row table and takes 40 seconds. What is wrong and how do you fix it? 
SELECT * FROM orders WHERE YEAR(created_at) = 2023;
▾
❌ The Problem
Function on indexed column breaks index usage. YEAR(created_at) forces MySQL to evaluate every row — full table scan on 50M rows.
⚠️ Common Wrong Fix
-- ❌ Wrong date format (MySQL uses YYYY-MM-DD)
WHERE created_at BETWEEN "01-01-2023" AND "31-12-2023"

-- ❌ BETWEEN misses records after 2023-12-31 00:00:00
-- e.g. 2023-12-31 14:35:22 is NOT included!
✅ Correct Fix
SELECT * FROM orders
WHERE created_at >= '2023-01-01'
  AND created_at < '2024-01-01';
-- Captures every millisecond of 2023 ✅
-- Index on created_at is fully utilized ✅
Rule: Never wrap an indexed column in a function. Always use range conditions with correct YYYY-MM-DD format on DATETIME columns.
SCORE
6/10 — Right problem, wrong date format
02
🟡 EXPLAIN & Composite Index
Table has index on (status, created_at). What does EXPLAIN show and will it use the index? 
EXPLAIN SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at DESC LIMIT 10;
▾
✅ Will It Use the Index?

Yes — and very efficiently. Composite index follows left-prefix rule:

  • status = 'pending' → equality on 1st column ✅
  • ORDER BY created_at DESC → sort on 2nd column ✅
  • MySQL walks index → rows already sorted → no extra sort needed
📊 Expected EXPLAIN Output
ColumnExpected ValueMeaning
typerefIndex lookup on status
keyyour index nameConfirms index is used
rowssmall estimateRows MySQL expects to read
Extranothing badNo filesort, no full scan ✅
❌ What is Using filesort?
Common misconception: It does NOT mean sorting to disk. It means MySQL had to do a separate sorting pass after fetching rows because the index couldn't provide pre-sorted data. 
-- When would filesort appear here?
-- If you use IN() instead of =
WHERE status IN ('pending', 'failed') ORDER BY created_at DESC
-- IN() = range → loses sort optimization → Using filesort ❌
Rule: = (equality) can use index for filter AND sort. IN(), >, BETWEEN (range) can filter using index but LOSE the sort optimization.
SCORE
5/10 — Index concept correct, EXPLAIN output unknown
03
🟡 Deadlock Detection & Fix
Two transactions updating the same rows in opposite order cause timeouts. Why is this a deadlock, how do you detect it, and how do you fix it?
▾
📋 Step-by-Step Deadlock
T=1  Session A → locks row id=1 ✅
T=2  Session B → locks row id=2 ✅
T=3  Session A → wants id=2 ❌ BLOCKED (B holds it)
T=4  Session B → wants id=1 ❌ BLOCKED (A holds it)
Result: Circular wait → DEADLOCK 💀
MySQL kills the cheaper transaction and returns:
ERROR 1213: Deadlock found when trying to get lock
🔍 Detection Commands
-- See the LAST deadlock in detail
SHOW ENGINE INNODB STATUS\G
-- Look for: LATEST DETECTED DEADLOCK section

-- See who is waiting RIGHT NOW
SELECT * FROM performance_schema.data_lock_waits;

-- See what locks are held
SELECT * FROM performance_schema.data_locks;
✅ 4 Fix Strategies

Fix 1 — Consistent Ordering

-- Always lock rows in same order: id=1 first, then id=2
-- Breaks the circular wait ✅

Fix 2 — Keep Transactions Short

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT; -- commit as fast as possible

Fix 3 — Lock Timeout

SET innodb_lock_wait_timeout = 5; -- kill after 5 seconds

Fix 4 — SELECT FOR UPDATE

BEGIN;
-- Lock both rows upfront in consistent order
SELECT * FROM accounts WHERE id IN (1,2) ORDER BY id FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
SCORE
4/10 — Concept known, detection & full fixes missed
04
🔴 Replication Lag
Your MySQL replica is lagging 45 minutes behind the master. What are the causes, how do you diagnose it right now, and how do you fix it?
▾
⚙️ How Replication Works
Master  → writes to Binary Log (binlog)
Replica → IO Thread copies binlog → Relay Log
Replica → SQL Thread replays relay log → applies changes
Lag = SQL Thread replaying slower than master is writing
⚠️ Causes of Replica Lag
CauseWhy It Happens
Single-threaded SQL threadMaster writes parallel, replica replays one-by-one
Heavy queries on master1 big DELETE takes 10min → replica also takes 10min
Network issuesSlow binlog transfer from master
Weaker replica hardwareLess CPU/RAM/disk than master
Missing indexes on replicaSame query slower on replica
Long running transactionsBlocks the SQL thread entirely
🔍 Diagnose Right Now
-- Run this on the REPLICA
SHOW REPLICA STATUS\G

-- Key fields to check:
Seconds_Behind_Master: 2700  ← 45 min confirmed
SQL_Running: Yes             ← SQL thread alive?
IO_Running: Yes              ← IO thread alive?
Last_Error: ...              ← blocking error?
-- NULL = replication broken entirely, not just lagging!
✅ Fix Strategies
-- Fix 1: Enable Parallel Replication (my.cnf on replica)
slave_parallel_workers = 4
slave_parallel_type = LOGICAL_CLOCK

-- Fix 2: Find stuck query
SHOW PROCESSLIST; -- look for "system user"

-- Fix 3: Batch large deletes on master
DELETE FROM logs WHERE created_at < '2023-01-01' LIMIT 1000;
-- Loop with sleep(0.1) between batches

-- Fix 4: Emergency — remove replica from load balancer
-- Stop sending reads to it until it catches up
SCORE
0/10 — Unknown topic, study immediately
05
🔴 Query Optimization & Architecture
This query on a 100M row table is causing 100% CPU spike. Find every problem and propose an architectural fix. 
SELECT user_id, COUNT(*) as total_orders, SUM(amount) as revenue
FROM orders WHERE status != 'cancelled'
GROUP BY user_id HAVING revenue > 10000 ORDER BY revenue DESC;
▾
❌ Every Problem
Problem 1: status != 'cancelled' — negative condition. MySQL can't use index efficiently. Likely full scan.
Problem 2: GROUP BY user_id on 90M+ rows. Aggregates everything before filtering.
Problem 3: HAVING instead of WHERE. Filters after aggregation — can't reduce rows early.
Problem 4: No covering index. MySQL hits actual table rows for every amount — massive disk IO.
📐 Index Strategy
-- ❌ Wrong order
INDEX (status, user_id, amount) -- status with != is weak as leader

-- ✅ Covering index — better order
INDEX (user_id, status, amount)
-- GROUP BY user_id benefits from leading column
-- status filters within each user partition
-- amount included → no table lookup = covering index
✅ Rewrite Query
SELECT user_id,
       COUNT(*) as total_orders,
       SUM(amount) as revenue
FROM orders
WHERE status IN ('completed', 'pending', 'processing') -- positive list
GROUP BY user_id
HAVING SUM(amount) > 10000
ORDER BY revenue DESC;
⭐ Architectural Fix — The Real Answer
Root cause: This is an OLAP query (analytics) running on an OLTP table (transactions). That's the architecture problem. 
-- Fix 1: Summary Table (run via scheduled job)
CREATE TABLE user_order_summary (
    user_id       INT PRIMARY KEY,
    total_orders  INT,
    total_revenue DECIMAL(15,2),
    last_updated  DATETIME
);
-- Dashboard query: milliseconds on thousands of rows ✅

-- Fix 2: Read Replica for analytics
-- Fix 3: Cache result in Redis every 15 minutes
SCORE
3/10 — Index instinct there, architecture missed
06
🔴 Table Partitioning
You have a 500M row transactions table growing 2M rows/day. Should you partition it? What type, and what are the hidden risks?
▾
📋 Partition Types
TypeHow It WorksBest For
RANGERows go into partition based on column rangeDates, years ✅
LISTRows go into partition based on exact valuesCountry, status
HASHRows distributed evenly by hashEven distribution
KEYMySQL manages the hashPrimary key splits
✅ Correct Implementation
-- RANGE by MONTH (better than year for 2M/day)
CREATE TABLE transactions (
    id         BIGINT NOT NULL,
    user_id    INT NOT NULL,
    amount     DECIMAL(15,2),
    created_at DATETIME NOT NULL
)
PARTITION BY RANGE (TO_DAYS(created_at)) (
    PARTITION p202401 VALUES LESS THAN (TO_DAYS('2024-02-01')),
    PARTITION p202402 VALUES LESS THAN (TO_DAYS('2024-03-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
❌ Hidden Risks (Most Candidates Miss)
Risk 1 — Unique Key Must Include Partition Key:
UNIQUE KEY (id) ❌ fails. Must be UNIQUE KEY (id, created_at) ✅
Risk 2 — Pruning Only on Partition Key:
WHERE user_id = 1234 still scans ALL partitions! You must filter on created_at.
Risk 3 — No Foreign Keys: Partitioned tables cannot have FK constraints.
Risk 4 — Function Kills Pruning:
WHERE YEAR(created_at) = 2024 ❌ — No pruning!
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01' ✅
⭐ Biggest Benefit (Rarely Mentioned):
ALTER TABLE transactions DROP PARTITION p2021;
Instant deletion of millions of old rows. vs. DELETE which takes hours!
SCORE
2/10 — Concept known, zero depth on risks

🔴 Backup & Disaster Recovery
3 AM. A developer ran UPDATE users SET password='hacked123' without a WHERE clause — 2 million users affected. What do you do?
▾
❌ Why Simple Rollback Won't Work
ROLLBACK only works if the transaction is still OPEN. If the developer ran UPDATE and it auto-committed, the transaction is closed — there is NOTHING to rollback. This is a Point-in-Time Recovery (PITR) situation.
🚨 First 5 Minutes
Minute 1 → Block all user logins (stop the bleeding)
Minute 2 → Put app in maintenance mode (stop writes)
Minute 3 → Get EXACT timestamp of bad UPDATE from developer
Minute 4 → Confirm you have backup + binlogs enabled
Minute 5 → Start recovery, inform management

REVOKE ALL ON users FROM 'app_user'@'%'; -- restrict access
🔧 Point-in-Time Recovery
-- Step 1: Restore last full backup (e.g. midnight backup)
mysql -u root -p < backup_midnight.sql
-- OR using XtraBackup (faster for large DBs)
xtrabackup --prepare --target-dir=/backup/midnight/
xtrabackup --copy-back --target-dir=/backup/midnight/

-- Step 2: Find the bad query in binlog
mysqlbinlog /var/lib/mysql/mysql-bin.000123 | grep -i "UPDATE users"

-- Step 3: Replay binlogs UP TO moment before disaster
mysqlbinlog /var/lib/mysql/mysql-bin.000123 \
  --start-datetime="2024-01-15 00:00:00" \
  --stop-datetime="2024-01-15 03:17:22" \
  | mysql -u root -p

-- Step 4: Verify
SELECT COUNT(*) FROM users WHERE password = 'hacked123';
-- Must return 0 ✅
✅ Prevention
-- Safe mode: forces WHERE clause on UPDATE/DELETE
SET sql_safe_updates = 1;

-- Always SELECT before UPDATE to see impact
SELECT COUNT(*) FROM users; -- check row count first

-- Binlog settings in my.cnf (MUST always be ON)
log_bin = ON
binlog_format = ROW      -- logs actual row values, not just SQL
expire_logs_days = 7    -- keep 7 days of binlogs
3-2-1 Backup Rule: 3 copies of data, 2 on different storage types, 1 offsite (S3/GCS). Test your backups monthly — a backup you've never restored is not a backup.
SCORE
1/10 — Most dangerous gap. Study PITR immediately.
⚡ Trap Questions — One Word / Short Answers

These are trick questions interviewers ask to catch you off-guard. Each has a short definitive answer. Click to reveal.

T1
⚡ Trap — Rollback
Can you ROLLBACK a DROP TABLE statement in MySQL?
▾
ANSWER No
DROP TABLE is a DDL statement. DDL statements in MySQL cause an implicit COMMIT before and after they execute. You cannot rollback DDL — the table is gone permanently. This is why we need backups.
T2
⚡ Trap — NULL Comparison
Will this query return rows where age is NULL? 
SELECT * FROM users WHERE age != 25;
▾
ANSWER No
NULL is never equal to anything — not even NULL. Any comparison with NULL returns UNKNOWN, not TRUE. To include NULLs you must explicitly add OR age IS NULL. This is one of the most common production bugs.
-- ❌ Misses NULL rows
WHERE age != 25

-- ✅ Includes NULL rows
WHERE age != 25 OR age IS NULL
T3
⚡ Trap — Index on Low Cardinality
Should you add an index on a status column that has only 3 values: 'active', 'inactive', 'pending'?
▾
ANSWER Depends
Low cardinality index alone is often useless — MySQL may ignore it and prefer a full scan. BUT as part of a composite index where status is the leading column with equality, it can be very effective. Context is everything.
Rule: Standalone index on 3-value column — probably useless. Composite index (status, created_at) — very powerful for filtered queries.
T4
⚡ Trap — COUNT Performance
Which is faster on a 50M row InnoDB table? 
SELECT COUNT(*) FROM orders;
SELECT COUNT(id) FROM orders;
▾
ANSWER Same
COUNT(*) and COUNT(id) are effectively the same speed in MySQL — the optimizer treats them identically. The real trap: InnoDB does NOT store row counts like MyISAM, so BOTH are slow on large tables without a covering index. Use a summary table for dashboard counts.
T5
⚡ Trap — TRUNCATE vs DELETE

T6
⚡ Trap — Index Usage
You have an index on email. Will this query use it? 
SELECT * FROM users WHERE email LIKE '%gmail.com';
▾
ANSWER No
A leading wildcard LIKE '%...' cannot use a B-Tree index because MySQL doesn't know where to start in the index. It forces a full table scan. Only trailing wildcards LIKE 'john%' can use the index.
-- ❌ Full scan — leading wildcard
WHERE email LIKE '%gmail.com'

-- ✅ Uses index — trailing wildcard
WHERE email LIKE 'john%'
T7
⚡ Trap — Transaction Isolation
What is MySQL's default transaction isolation level?
▾
ANSWER REPEATABLE READ
Most databases default to READ COMMITTED — MySQL defaults to REPEATABLE READ. This is a common interview trap. REPEATABLE READ prevents dirty reads and non-repeatable reads but can still have phantom reads (though InnoDB prevents phantoms with gap locks).
