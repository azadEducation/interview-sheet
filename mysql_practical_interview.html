SQL Query Execution Order
P1In what order does MySQL actually execute a SELECT statement?Pro▾
Answer
This is one of the most misunderstood fundamentals. The order you write SQL is not the order it executes. Understanding this explains WHY certain errors happen and WHY aliases work in some clauses but not others.
Execution Order (Logical)
StepClauseWhat HappensPractical Impact
1FROMSource tables identifiedTable access, storage engine called
2ONJOIN condition evaluatedFilters rows during join (faster than WHERE for outer joins)
3JOINTables mergedINNER/LEFT/RIGHT applied
4WHERERow-level filterCannot use SELECT aliases here (aliases not defined yet!)
5GROUP BYRows groupedCreates groups for aggregation
6HAVINGGroup-level filterCan use aggregate functions
7SELECTColumns computedAliases defined HERE
8DISTINCTDuplicates removedAfter SELECT runs
9ORDER BYResults sortedCAN use SELECT aliases (defined in step 7)
10LIMIT/OFFSETRows slicedVery last step
-- WHY this fails (alias in WHERE):
SELECT salary * 1.1 AS new_salary
FROM employees
WHERE new_salary > 50000; -- ❌ ERROR: Unknown column 'new_salary'
-- WHERE runs at step 4, alias defined at step 7

-- Fix 1: Repeat the expression
WHERE salary * 1.1 > 50000;

-- Fix 2: Subquery / CTE
WITH calc AS (SELECT salary * 1.1 AS new_salary FROM employees)
SELECT * FROM calc WHERE new_salary > 50000;

-- WHY this WORKS (alias in ORDER BY):
SELECT first_name || ' ' || last_name AS full_name
FROM employees
ORDER BY full_name; -- ✅ Works! ORDER BY is step 9, alias defined step 7
🎯 Interview answer tip: Most candidates write SQL but can't explain execution order. Nail this and you instantly stand out. The interviewer will follow up with "then why does GROUP BY alias work in MySQL?" — because MySQL extends the standard and resolves GROUP BY aliases as a special case.
P2What is the difference between ON and WHERE in a JOIN?Hard▾
Answer
For INNER JOINs, ON and WHERE produce identical results. For OUTER JOINs, they behave very differently because ON filters during the join (step 2) while WHERE filters after (step 4).
-- LEFT JOIN: ON filters during join (keeps all left rows)
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'completed';
-- Result: ALL users, only completed orders (NULLs for users without completed orders)

-- LEFT JOIN: WHERE filters AFTER join (converts to INNER JOIN effectively)
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'completed';
-- Result: only users who HAVE completed orders (NULL rows eliminated by WHERE)
⚠️ Classic interview trap: "Write a query to get all users and their completed orders if any." Wrong: WHERE clause. Right: ON clause with LEFT JOIN. Mixing these up is the #1 JOIN mistake in interviews.

C1Find duplicate emails and delete keeping only the lowest IDHard▾
🎯 LeetCode #196 — Top Company Question
-- Table: users (id, email)

-- Step 1: Find duplicates
SELECT email, COUNT(*) cnt
FROM users
GROUP BY email
HAVING cnt > 1;

-- Step 2: Delete duplicates, keep lowest id
-- Method 1: Self-join (works in all versions)
DELETE u1
FROM users u1
INNER JOIN users u2
  ON u1.email = u2.email
  AND u1.id > u2.id; -- keep the lower id

-- Method 2: Subquery (readable)
DELETE FROM users
WHERE id NOT IN (
  SELECT MIN(id) FROM users GROUP BY email
);
-- Note: MySQL doesn't allow direct subquery on same table in DELETE
-- Wrap in another subquery to bypass:
DELETE FROM users WHERE id NOT IN (
  SELECT id FROM (SELECT MIN(id) id FROM users GROUP BY email) t
);
💡 Key MySQL quirk: MySQL prohibits modifying a table in a subquery that's directly referenced in the same DELETE/UPDATE. The double-subquery trick (wrapping in an extra SELECT) forces MySQL to create a temp table, bypassing this restriction.
C2Consecutive login streak — find users with 3+ consecutive days logged inPro▾
🎯 Facebook / Meta Interview Question
-- Table: logins (user_id, login_date)
-- Technique: row_number trick for consecutive dates

WITH dated AS (
  SELECT
    user_id,
    login_date,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date) rn
  FROM (
    SELECT DISTINCT user_id, login_date FROM logins
  ) d
),
grouped AS (
  SELECT
    user_id,
    login_date,
    -- Subtract row_number from date to get a "group key"
    -- Consecutive dates share the same group_key!
    DATE_SUB(login_date, INTERVAL rn DAY) AS group_key
  FROM dated
)
SELECT DISTINCT user_id
FROM grouped
GROUP BY user_id, group_key
HAVING COUNT(*) >= 3;  -- 3+ consecutive days
🧠 The trick: If dates are consecutive, (date - row_number) gives the same constant value. This is the canonical "island detection" technique — memorize it, it appears in interviews constantly.
C3Find the median salary — without using any aggregate function directlyPro▾
🎯 Amazon / Goldman Sachs Interview
-- Method 1: MySQL 8 with PERCENTILE_CONT equivalent
WITH ranked AS (
  SELECT
    salary,
    ROW_NUMBER() OVER (ORDER BY salary) rn,
    COUNT(*) OVER() total
  FROM employees
)
SELECT AVG(salary) AS median
FROM ranked
WHERE rn BETWEEN total/2.0 AND total/2.0 + 1;
-- AVG handles both odd (1 middle) and even (2 middles) counts

-- Method 2: Classic without window functions
SELECT AVG(salary) median
FROM (
  SELECT salary
  FROM employees
  ORDER BY salary
  LIMIT 2 -- handles both odd and even
  OFFSET (SELECT (COUNT(*) - 1) / 2 FROM employees)
) m;
C4Department-wise top 3 earners (handle ties correctly)Hard▾
🎯 LeetCode #185 — Uber / Microsoft
-- Tables: employees(id, name, salary, dept_id), departments(id, name)

WITH ranked AS (
  SELECT
    e.name,
    e.salary,
    d.name AS dept,
    DENSE_RANK() OVER (
      PARTITION BY e.dept_id
      ORDER BY e.salary DESC
    ) dr
  FROM employees e
  JOIN departments d ON e.dept_id = d.id
)
SELECT dept, name, salary
FROM ranked
WHERE dr <= 3
ORDER BY dept, salary DESC;
⚠️ Use DENSE_RANK, not RANK: If two people tie for 2nd, RANK skips 3rd (gives 2,2,4). DENSE_RANK gives (2,2,3) — both show as top 3. RANK would incorrectly exclude the true 3rd place. Interviewers test this distinction deliberately.

🎯 Product Analytics — Google / Flipkart
-- Table: purchases(user_id, product_name)

-- Method 1: NOT EXISTS (usually fastest)
SELECT DISTINCT user_id
FROM purchases
WHERE product_name = 'ProductA'
  AND NOT EXISTS (
    SELECT 1 FROM purchases p2
    WHERE p2.user_id = purchases.user_id
      AND p2.product_name = 'ProductB'
  );

-- Method 2: LEFT JOIN anti-join (often preferred by interviewers)
SELECT a.user_id
FROM purchases a
LEFT JOIN purchases b
  ON a.user_id = b.user_id AND b.product_name = 'ProductB'
WHERE a.product_name = 'ProductA'
  AND b.user_id IS NULL;

-- Method 3: Conditional aggregation (elegant)
SELECT user_id
FROM purchases
GROUP BY user_id
HAVING
  SUM(CASE WHEN product_name='ProductA' THEN 1 ELSE 0 END) > 0
  AND SUM(CASE WHEN product_name='ProductB' THEN 1 ELSE 0 END) = 0;
C7Find gaps in an auto-increment ID sequenceHard▾
🎯 Data Integrity — Backend Engineering
-- Table: orders(id, ...) — find missing IDs (gaps)

-- Method 1: Self-join (MySQL 5.x compatible)
SELECT a.id + 1 AS gap_start
FROM orders a
WHERE NOT EXISTS (
  SELECT 1 FROM orders b WHERE b.id = a.id + 1
)
AND a.id < (SELECT MAX(id) FROM orders);

-- Method 2: LAG window function (MySQL 8, clean)
WITH gaps AS (
  SELECT
    id,
    LAG(id) OVER (ORDER BY id) prev_id
  FROM orders
)
SELECT
  prev_id + 1 AS gap_from,
  id - 1      AS gap_to,
  id - prev_id - 1 AS missing_count
FROM gaps
WHERE id > prev_id + 1;

C8Retention analysis — what % of users returned after Day 1?Pro▾
🎯 Growth Analytics — Startup / Big Tech
-- Table: events(user_id, event_date)

WITH first_day AS (
  SELECT user_id, MIN(event_date) signup_date
  FROM events
  GROUP BY user_id
),
returned AS (
  SELECT f.user_id
  FROM first_day f
  JOIN events e ON f.user_id = e.user_id
    AND e.event_date = DATE_ADD(f.signup_date, INTERVAL 1 DAY)
)
SELECT
  COUNT(DISTINCT r.user_id) day1_retained,
  COUNT(DISTINCT f.user_id) total_users,
  ROUND(COUNT(DISTINCT r.user_id) * 100.0 / COUNT(DISTINCT f.user_id), 2) day1_retention_pct
FROM first_day f
LEFT JOIN returned r ON f.user_id = r.user_id;
C9Write a query to detect billing fraud: same card, different users within 10 minutesPro▾
🎯 Fintech / Payment Systems Interview
-- Table: transactions(id, user_id, card_last4, txn_time, amount)

SELECT
  t1.card_last4,
  t1.user_id  AS user1,
  t2.user_id  AS user2,
  t1.txn_time AS time1,
  t2.txn_time AS time2,
  TIMESTAMPDIFF(SECOND, t1.txn_time, t2.txn_time) gap_secs
FROM transactions t1
JOIN transactions t2
  ON  t1.card_last4 = t2.card_last4
  AND t1.user_id   != t2.user_id
  AND t2.txn_time  >  t1.txn_time
  AND TIMESTAMPDIFF(SECOND, t1.txn_time, t2.txn_time) <= 600
ORDER BY t1.card_last4, t1.txn_time;
🧠 Performance note: This self-join is expensive on large tables. In production, add index on (card_last4, txn_time) and run in time-windowed batches, not on the full history. This is worth discussing with the interviewer.
C10Write an UPSERT and explain INSERT ... ON DUPLICATE KEY UPDATEMedium▾
🎯 Very Common in Backend Interviews
-- UPSERT: insert if not exists, update if exists

-- Method 1: ON DUPLICATE KEY UPDATE
INSERT INTO user_stats (user_id, login_count, last_seen)
VALUES (42, 1, NOW())
ON DUPLICATE KEY UPDATE
  login_count = login_count + 1,
  last_seen = NOW();

-- Method 2: REPLACE INTO (deletes + reinserts — different ID!)
REPLACE INTO user_stats (user_id, login_count, last_seen)
VALUES (42, 1, NOW());
-- ⚠️ REPLACE deletes the old row first, triggers DELETE triggers!
-- ⚠️ FK references to old row will fail with ON DELETE RESTRICT

-- Method 3: INSERT IGNORE (skip on duplicate, no update)
INSERT IGNORE INTO user_stats (user_id, login_count)
VALUES (42, 1);
-- Silently ignores duplicate key errors (and other errors too!)

-- Best practice: ON DUPLICATE KEY UPDATE
-- Atomic, fires only UPDATE triggers on conflict, preserves ID

-- Bulk upsert
INSERT INTO product_inventory (product_id, qty, updated_at)
VALUES
  (1, 100, NOW()),
  (2, 50,  NOW()),
  (3, 75,  NOW())
ON DUPLICATE KEY UPDATE
  qty = VALUES(qty),   -- deprecated in 8.0, use aliases below
  updated_at = VALUES(updated_at);

-- MySQL 8.0.19+ alias syntax (preferred)
INSERT INTO product_inventory (product_id, qty)
VALUES (1, 100) AS new_vals
ON DUPLICATE KEY UPDATE qty = new_vals.qty;
C11Write a query to transpose rows to columns dynamicallyPro▾
🎯 Reporting — Data Engineering
-- Static pivot (known categories)
SELECT
  student_id,
  MAX(CASE WHEN subject='Math'    THEN score END) Math,
  MAX(CASE WHEN subject='Science' THEN score END) Science,
  MAX(CASE WHEN subject='English' THEN score END) English
FROM scores
GROUP BY student_id;

-- Dynamic pivot (unknown categories) — using prepared statements
SET @sql = NULL;

SELECT GROUP_CONCAT(
  DISTINCT CONCAT(
    "MAX(CASE WHEN subject='", subject,
    "' THEN score END) AS `", subject, "`"
  )
) INTO @sql
FROM scores;

SET @sql = CONCAT('SELECT student_id, ', @sql, ' FROM scores GROUP BY student_id');
PREPARE stmt FROM @sql;
EXECUTE stmt;
DEALLOCATE PREPARE stmt;
C12Running total with reset condition (per month)Hard▾
🎯 Financial Reporting Interview
-- Running total of sales that resets every month
SELECT
  sale_date,
  amount,
  DATE_FORMAT(sale_date, '%Y-%m') AS month,
  SUM(amount) OVER (
    PARTITION BY DATE_FORMAT(sale_date, '%Y-%m')  -- reset per month
    ORDER BY sale_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS monthly_running_total
FROM sales
ORDER BY sale_date;
💡 Frame clause: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is the default for ORDER BY in window functions, but being explicit shows mastery. The alternative RANGE BETWEEN handles ties differently (includes all tied rows in the frame).

C15Hierarchical query: print org chart with levels and pathPro▾
🎯 Complex — Shows Recursive CTE Mastery
-- Table: employees(id, name, manager_id)

WITH RECURSIVE org_chart AS (
  -- Anchor: CEO (no manager)
  SELECT
    id,
    name,
    manager_id,
    0    AS level,
    name AS path,
    CAST(id AS CHAR(1000)) AS id_path
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive: children
  SELECT
    e.id,
    e.name,
    e.manager_id,
    o.level + 1,
    CONCAT(o.path, ' → ', e.name),
    CONCAT(o.id_path, ',', e.id)
  FROM employees e
  JOIN org_chart o ON e.manager_id = o.id
)
SELECT
  REPEAT('  ', level) || name AS org_tree,
  level,
  path
FROM org_chart
ORDER BY id_path;  -- Preserves hierarchy order

E1Read this EXPLAIN output — what's wrong and how to fix it?Pro▾
Scenario
EXPLAIN SELECT * FROM orders
WHERE DATE(created_at) = '2024-01-15'\G
id: 1    select_type: SIMPLE    table: orders    type: ALL    key: NULL    rows: 8,432,000    Extra: Using where
Diagnosis: 3 critical problems
❌ What's Wrong
1. type: ALL → full table scan on 8.4M rows
2. key: NULL → no index used
3. DATE(created_at) → function wraps indexed column, destroying index usage
✅ The Fix
-- Remove function, use range
WHERE created_at
  BETWEEN '2024-01-15 00:00:00'
  AND     '2024-01-15 23:59:59';
After Fix — EXPLAIN shows:
type: range    key: idx_created_at    rows: 1,247    Extra: Using index condition
✅ Result: 8.4M row scan → 1,247 rows. Query time: 8 seconds → 12ms. This is a 650x improvement.
E2What does "Using filesort" mean and how to eliminate it?Hard▾
Answer
"Using filesort" means MySQL cannot use an index to satisfy ORDER BY. It must sort the result set in memory (or a temp file if too large). This is always a red flag to investigate.
-- ❌ Triggers filesort
SELECT * FROM orders
WHERE user_id = 42
ORDER BY created_at DESC;
-- Index on user_id exists but not on created_at too

-- ✅ Fix: composite index matching WHERE + ORDER BY
CREATE INDEX idx_user_date
  ON orders (user_id, created_at DESC);  -- MySQL 8: true DESC index

-- ✅ Fix 2: Ensure sort buffer is large enough
sort_buffer_size = 4M  -- if must sort, at least do it in memory

-- EXPLAIN ANALYZE (MySQL 8): shows actual sort time
EXPLAIN ANALYZE SELECT * FROM orders
WHERE user_id = 42 ORDER BY created_at DESC\G
⚠️ Composite index rule: The ORDER BY columns must come after the WHERE equality columns in the composite index, and the ORDER BY direction must match the index direction. (user_id, created_at DESC) serves WHERE user_id=? ORDER BY created_at DESC perfectly.
E3What does "Using temporary" mean and why is it worse than filesort?Pro▾
Answer
"Using temporary" means MySQL must create an internal temporary table to process the query. This happens with GROUP BY on non-indexed columns, UNION without index, or complex subqueries.
-- Triggers "Using temporary; Using filesort"
SELECT status, COUNT(*) FROM orders GROUP BY status ORDER BY COUNT(*);

-- Check if temp table went to disk
SHOW STATUS LIKE 'Created_tmp_disk_tables';
SHOW STATUS LIKE 'Created_tmp_tables';
-- Ratio of disk/total should be < 5%

-- Tune to keep temp tables in memory:
tmp_table_size     = 256M
max_heap_table_size = 256M  -- both must be set
Extra ValueSeverityMeaning
Using index✅ BestCovering index, no table access
Using where✅ GoodFilter applied after index scan
Using index condition✅ GoodICP: filter pushed into storage engine
Using filesort⚠️ InvestigateSort needed, no index for ORDER BY
Using temporary❌ BadTemp table created
Using join buffer (BNL)❌ BadNo index on joined column
Full scan on NULL key❌ Very badNULL comparison kills index
E4What is Index Condition Pushdown (ICP) and why does it matter?Pro▾
Answer
ICP is an optimization where MySQL pushes WHERE conditions into the storage engine (InnoDB) during index scan, instead of fetching rows then filtering at the MySQL layer. EXPLAIN shows "Using index condition".
-- Without ICP (old behavior):
-- 1. InnoDB scans idx_lastname for lastname='Smith'
-- 2. Fetches full rows for each match
-- 3. MySQL layer checks firstname LIKE 'J%'
-- Many rows fetched unnecessarily!

-- With ICP (MySQL 5.6+):
-- 1. InnoDB scans idx on (lastname, firstname)
-- 2. Checks BOTH conditions IN the storage engine
-- 3. Only fetches rows passing BOTH filters
-- Fewer rows fetched = less I/O

SELECT * FROM employees
WHERE lastname = 'Smith'
  AND firstname LIKE 'J%';
-- Extra: Using index condition  ← ICP active

-- Check/toggle ICP:
SET optimizer_switch = 'index_condition_pushdown=on'; -- default ON

Index Strategy — How Seniors Think
I1How do you decide WHICH columns to index on a new table?Pro▾
Senior DBA Decision Framework
  1. Always index the PRIMARY KEY (clustered, automatic)
  2. Index every FOREIGN KEY column — MySQL doesn't do this automatically. Missing FK indexes = full scan on every JOIN and every parent DELETE
  3. Identify your most frequent query patterns from application code or slow query log
  4. For each query: columns in WHERE (high-cardinality first), then JOIN ON, then ORDER BY, then GROUP BY
  5. Check cardinality: COUNT(DISTINCT col) / COUNT(*) — if < 5%, reconsider
  6. Consider composite indexes: group related WHERE + ORDER BY columns together
  7. Think about covering indexes for critical hot paths (SELECT only indexed columns)
  8. Never index a column that's modified very frequently — index maintenance cost
-- FK index audit: find FK columns without an index
SELECT
  kcu.table_name,
  kcu.column_name AS fk_column,
  kcu.referenced_table_name
FROM information_schema.KEY_COLUMN_USAGE kcu
LEFT JOIN information_schema.STATISTICS s
  ON  s.table_name   = kcu.table_name
  AND s.column_name  = kcu.column_name
  AND s.table_schema = kcu.table_schema
WHERE kcu.referenced_table_name IS NOT NULL
  AND s.index_name IS NULL  -- no index on FK!
  AND kcu.table_schema = 'your_db';
🔬 25-year rule: The single most common performance problem I've seen in production is missing indexes on foreign key columns. Every experienced DBA has seen MySQL do a 10-second full scan on a child table because the FK column had no index.
I2What is index merge and when does MySQL use it?Pro▾
Answer
Index merge allows MySQL to use multiple indexes on the same table in one query and merge the results. Sounds good, often isn't.
SELECT * FROM orders
WHERE user_id = 42 OR status = 'urgent';

-- MySQL might use: Extra = "Using union(idx_user,idx_status); Using where"
-- It scans idx_user for user_id=42, scans idx_status for status='urgent'
-- Then does a UNION of row IDs (sort + merge)
⚠️ Index merge is often a symptom of bad schema design: If you see index merge frequently, it usually means you need a better composite index. Index merge has merge overhead. A well-designed composite index is usually faster.
-- Better: composite index
CREATE INDEX idx_user_status ON orders(user_id, status);

-- Or rewrite OR as UNION ALL (often faster)
SELECT * FROM orders WHERE user_id = 42
UNION ALL
SELECT * FROM orders WHERE status = 'urgent' AND user_id != 42;
I3How many indexes is "too many"? What's the hidden cost?Pro▾
Answer
Every index you add has ongoing costs that compound over time. There's no magic number, but 5-7 indexes per table is usually a soft ceiling before you investigate overlap.
Index CostImpact
Write overheadEvery INSERT, UPDATE, DELETE must update ALL indexes. 10 indexes = 10x write cost
StorageEach index is a separate B+Tree. On 100GB table, indexes can be another 50-200GB
Buffer pool pressureIndex pages compete with data pages in the buffer pool
DDL timeAdding a new index on 500GB table = hours
Optimizer confusionToo many choices = optimizer sometimes picks wrong one
-- Find unused indexes (MySQL 8 performance_schema)
SELECT
  t.object_schema,
  t.object_name AS table_name,
  t.index_name
FROM performance_schema.table_io_waits_summary_by_index_usage t
WHERE t.index_name IS NOT NULL
  AND t.count_star = 0  -- never used since last restart
  AND t.object_schema = 'your_db'
  AND t.index_name != 'PRIMARY'
ORDER BY t.object_name, t.index_name;

-- Make index INVISIBLE first (MySQL 8) — test before dropping
ALTER TABLE orders ALTER INDEX idx_old INVISIBLE;
-- Monitor for a week. If no problems → DROP INDEX

Schema Design — Patterns & Anti-Patterns
S1The EAV (Entity-Attribute-Value) pattern — why it's a trapPro▾
EAV stores dynamic attributes as rows rather than columns. Seductive for flexibility, catastrophic for performance.
❌ EAV Schema
-- Flexible but terrible
product_attributes
(product_id, attr_name, attr_value)

(1, 'color',  'red')
(1, 'size',   'XL')
(1, 'weight', '200g')
✅ Proper Schema
-- Type-safe, indexable, fast
products
(id, color, size, weight_g)

-- Or: JSON column for truly
-- dynamic attributes
(id, attributes JSON)
⚠️ EAV problems: Getting all attributes of one product requires N queries or a horrific pivot. You can't enforce data types. You can't index effectively. Querying "all red XL products" requires multiple self-joins. I've inherited EAV systems that took 45 seconds for basic product searches. The fix was a 3-month migration project.
S2Storing comma-separated IDs in a column — why it's always wrongMedium▾
❌ Anti-Pattern
-- users table
id | role_ids
1  | "2,5,8"   ← ❌
2  | "1,3"

-- Problems:
-- Cannot use FK constraints
-- LIKE '%5%' scans everything
-- Can't JOIN efficiently
-- '5' matches '15' or '52'!
-- Violates 1NF
✅ Correct: Junction Table
-- user_roles
user_id | role_id
1       | 2
1       | 5
1       | 8

CREATE INDEX idx_user (user_id);
CREATE INDEX idx_role (role_id);
-- Fast JOINs, FK constraints,
-- clean aggregations
⚠️ When it seems acceptable: Rarely, with JSON array and a generated column index. But even then, the junction table is almost always better for anything queried frequently.
S3Polymorphic associations — design pattern and its trade-offsHard▾
-- Comments can belong to: Post, Video, Product
-- Polymorphic approach:
CREATE TABLE comments (
  id             BIGINT PRIMARY KEY,
  body           TEXT,
  commentable_id BIGINT,
  commentable_type VARCHAR(50),  -- 'Post','Video','Product'
  created_at     DATETIME
);
-- ❌ Cannot add FK constraint (commentable_id refs multiple tables)
-- ❌ No referential integrity
-- ❌ JOIN requires knowing the type first

-- ✅ Alternative: Exclusive arc pattern (separate FK per type)
CREATE TABLE comments (
  id         BIGINT PRIMARY KEY,
  body       TEXT,
  post_id    BIGINT NULL REFERENCES posts(id),
  video_id   BIGINT NULL REFERENCES videos(id),
  product_id BIGINT NULL REFERENCES products(id),
  -- Constraint: exactly one must be non-null
  CHECK (
    (post_id IS NOT NULL) + (video_id IS NOT NULL) + (product_id IS NOT NULL) = 1
  )
);
💡 Production pick: Polymorphic without FK is fast to build but painful to maintain. Exclusive arc adds nullable columns but preserves referential integrity. For high-scale systems (millions of comments), a separate comments table per entity type scales best.
S4Soft delete design — the right and wrong wayHard▾
-- Common implementation (problematic)
ALTER TABLE users ADD COLUMN deleted_at DATETIME NULL;

-- Problems:
-- 1. Every query needs WHERE deleted_at IS NULL
-- 2. UNIQUE constraints include soft-deleted rows (email conflict!)
-- 3. Indexes bloated with "deleted" rows

-- ✅ Fix 1: Partial index (MySQL 8 functional index workaround)
-- Can't do WHERE in index, but: unique + soft delete solution:
CREATE UNIQUE INDEX idx_email_active
  ON users (email, deleted_at);  -- deleted_at=NULL treated as unique per row in InnoDB

-- ✅ Fix 2: Archive table pattern (better at scale)
CREATE TABLE users_deleted LIKE users;
ALTER TABLE users_deleted ADD COLUMN deleted_at DATETIME NOT NULL;

-- On delete: move to archive table
INSERT INTO users_deleted SELECT *, NOW() FROM users WHERE id = 42;
DELETE FROM users WHERE id = 42;
🔬 Scale reality: After 5 years, a soft-delete column means 40% of your table is "deleted" rows that every query must filter. Indexes include them. The archive table approach keeps your active table lean and fast.

Stored Procedures — Advanced Patterns
SP1Cursor-based processing with proper error handlingPro▾
DELIMITER //
CREATE PROCEDURE ProcessLargeTable()
BEGIN
  -- Declare variables
  DECLARE v_user_id   INT;
  DECLARE v_email     VARCHAR(255);
  DECLARE v_done      BOOLEAN DEFAULT FALSE;
  DECLARE v_err_code  CHAR(5) DEFAULT '00000';
  DECLARE v_err_msg   TEXT;
  DECLARE v_count     INT DEFAULT 0;

  -- Cursor declaration (must be before handlers)
  DECLARE cur CURSOR FOR
    SELECT id, email FROM users WHERE needs_processing = 1;

  -- Handler: set flag when cursor exhausted
  DECLARE CONTINUE HANDLER FOR NOT FOUND
    SET v_done = TRUE;

  -- Handler: catch SQL exceptions
  DECLARE CONTINUE HANDLER FOR SQLEXCEPTION
  BEGIN
    GET DIAGNOSTICS CONDITION 1
      v_err_code = RETURNED_SQLSTATE,
      v_err_msg  = MESSAGE_TEXT;
    INSERT INTO error_log (proc_name, error_code, error_msg, created_at)
    VALUES ('ProcessLargeTable', v_err_code, v_err_msg, NOW());
  END;

  OPEN cur;

  process_loop: LOOP
    FETCH cur INTO v_user_id, v_email;
    IF v_done THEN LEAVE process_loop; END IF;

    -- Process each row
    UPDATE users SET processed_at = NOW(), needs_processing = 0
    WHERE id = v_user_id;

    SET v_count = v_count + 1;

    -- Commit in batches to avoid long transactions
    IF v_count % 1000 = 0 THEN
      COMMIT;
    END IF;

  END LOOP;

  CLOSE cur;
  COMMIT;  -- final commit

  SELECT CONCAT('Processed: ', v_count) AS result;
END//
DELIMITER ;
⚠️ Production cursor advice: Cursors are slow (row-by-row = anti-set-based). Always prefer a single UPDATE...WHERE over a cursor when possible. Use cursors only when you genuinely need per-row logic that can't be expressed in set-based SQL.
SP2Dynamic SQL with PREPARE/EXECUTE — practical patternsHard▾
-- Generic: search any table, any column
DELIMITER //
CREATE PROCEDURE DynamicSearch(
  IN p_table   VARCHAR(64),
  IN p_column  VARCHAR(64),
  IN p_value   VARCHAR(255)
)
BEGIN
  SET @sql = CONCAT(
    'SELECT * FROM `',
    REPLACE(p_table, '`', ''),  -- sanitize!
    '` WHERE `',
    REPLACE(p_column, '`', ''),
    '` = ?'
  );
  PREPARE stmt FROM @sql;
  SET @val = p_value;
  EXECUTE stmt USING @val;  -- parameterized, safe!
  DEALLOCATE PREPARE stmt;
END//
DELIMITER ;
⚠️ SQL injection in procs: Even inside stored procedures, dynamic SQL via CONCAT is vulnerable. Always use PREPARE + USING for user values. Sanitize identifiers (table/column names) by stripping backticks and validating against information_schema.

CN1How do MySQL connections work internally?Hard▾
Each MySQL connection = one OS thread (or a thread from pool). Each thread uses memory for: connection buffer, sort buffer, join buffer, read buffer, etc. 1000 connections × 4MB per thread = 4GB RAM just for connections.
VariablePer-connection memoryDefaultGuidance
sort_buffer_sizeSort operations256KB1-4MB
join_buffer_sizeHash/BNL joins256KB1-4MB
read_buffer_sizeSequential scans128KB1MB
read_rnd_buffer_sizeRandom reads after sort256KB1MB
thread_stackThread stack1MBKeep default
-- Max RAM = innodb_buffer_pool + (max_connections × per_thread_memory)
-- Example: 32GB server
innodb_buffer_pool_size = 24G
max_connections         = 400
sort_buffer_size        = 1M   -- 400 × 1M = 400MB
join_buffer_size        = 1M   -- 400 × 1M = 400MB
-- Total: 24G + 1.6G overhead + 400M × 2 = ~26G ✅
⚠️ The "max_connections = 10000" mistake: Setting max_connections very high without accounting for per-thread memory causes OOM kills. 400 connections × all buffers often uses 4-8GB. Calculate first.

Interview Trap Questions — Don't Get Caught
T1What's the output of: SELECT NULL = NULL?Easy▾
SELECT NULL = NULL;   -- NULL (not TRUE, not FALSE!)
SELECT NULL != NULL;  -- NULL
SELECT NULL IS NULL; -- 1 (TRUE)  ← correct way
SELECT NULL <=> NULL;  -- 1 (NULL-safe equal operator)

-- NULL in WHERE:
SELECT * FROM users WHERE phone = NULL;    -- returns 0 rows! Always.
SELECT * FROM users WHERE phone IS NULL;   -- correct

-- NULL in aggregates:
SELECT COUNT(*) FROM users;          -- counts all rows
SELECT COUNT(phone) FROM users;      -- excludes NULLs!
SELECT SUM(amount) FROM orders;     -- NULLs ignored in SUM
SELECT 1 + NULL;                     -- NULL (NULL is contagious)
⚠️ NULL propagation: Any arithmetic with NULL = NULL. In GROUP BY, NULLs are grouped together. In ORDER BY, NULLs sort first (ASC) or last (DESC) — MySQL-specific behavior.
T2What happens when you GROUP BY without aggregating all non-grouped columns? (ONLY_FULL_GROUP_BY)Hard▾
-- This query: which name does it return?
SELECT dept_id, name, MAX(salary)
FROM employees
GROUP BY dept_id;
-- name is NOT in GROUP BY and NOT aggregated

-- MySQL 5.7+ with ONLY_FULL_GROUP_BY: ERROR 1055

-- Check sql_mode:
SELECT @@sql_mode;  -- Should contain ONLY_FULL_GROUP_BY

-- Correct ways:
-- Option 1: Include name in GROUP BY
SELECT dept_id, name, MAX(salary) FROM employees GROUP BY dept_id, name;

-- Option 2: Use ANY_VALUE() to explicitly pick any
SELECT dept_id, ANY_VALUE(name), MAX(salary) FROM employees GROUP BY dept_id;

-- Option 3: Get name of the highest earner (window function)
WITH ranked AS (
  SELECT dept_id, name, salary,
    ROW_NUMBER() OVER(PARTITION BY dept_id ORDER BY salary DESC) rn
  FROM employees
)
SELECT dept_id, name, salary FROM ranked WHERE rn=1;

T5Why does COUNT(*) differ from COUNT(column) and COUNT(DISTINCT col)?Medium▾
-- Sample data: users(id, email, phone)
-- id:    1,    2,    3,    4,    5
-- email: a,    b,    b,    c,    d   (b duplicate)
-- phone: X,    Y, NULL,    Y, NULL   (2 NULLs, Y duplicate)

SELECT
  COUNT(*)              AS c1,  -- 5  (all rows, NULLs included)
  COUNT(1)              AS c2,  -- 5  (same as COUNT(*))
  COUNT(phone)          AS c3,  -- 3  (excludes NULLs!)
  COUNT(DISTINCT email) AS c4,  -- 4  (excludes NULLs + deduplicates)
  COUNT(DISTINCT phone) AS c5   -- 2  (X, Y — NULLs excluded)
FROM users;
⚠️ Performance: COUNT(*) and COUNT(1) are identical — MySQL optimizes both. COUNT(DISTINCT col) on large tables without an index requires a full scan + sort. Add an index on the column if you COUNT(DISTINCT) it frequently.
T6What is the difference between TRUNCATE and DELETE with no WHERE on performance?Medium▾
-- Both remove all rows, but very differently internally:

-- DELETE without WHERE:
-- 1. Reads every row
-- 2. Writes to undo log (for rollback)
-- 3. Fires DELETE triggers
-- 4. Acquires row locks (or table lock for MyISAM)
-- 5. Generates binlog events for every row → massive binlog → replication lag
-- Time on 10M row table: minutes to hours

-- TRUNCATE:
-- 1. Drops and recreates the table (DDL operation)
-- 2. No undo log writing
-- 3. No triggers
-- 4. Resets AUTO_INCREMENT to 1
-- 5. Single binlog event
-- Time: milliseconds

DELETE FROM large_table;   -- ❌ slow on large tables
TRUNCATE TABLE large_table; -- ✅ instant, but no rollback
T7Explain what happens step-by-step when you run: UPDATE users SET balance=balance-100 WHERE id=5Pro▾
  1. Parser converts SQL to parse tree, checks syntax
  2. Optimizer finds best plan: likely index lookup on PRIMARY KEY id=5
  3. InnoDB checks if page with id=5 is in buffer pool. If not, reads from disk
  4. InnoDB acquires exclusive row lock (X-lock) on the row
  5. Reads current balance value from row
  6. Writes old row image to undo log (enables rollback + MVCC reads)
  7. Modifies the row in the buffer pool page (balance = balance - 100)
  8. Marks page as dirty in the flush list
  9. Writes to redo log buffer (WAL — before page hits disk)
  10. If autocommit=ON: immediately flushes redo log to disk (fsync)
  11. Returns "1 row affected" to client
  12. Background: page eventually flushed to data file (checkpoint)
  13. Background: binlog event written (after commit, for replication)
🔬 This answer demonstrates: Buffer pool, undo log, redo log, MVCC, WAL, locking — all in one query. This level of detail makes interviewers very impressed. Most candidates only know step 1-2.
T8What is the "Lost Update" problem and how does MySQL prevent it?Pro▾
-- Two transactions read balance=100, both subtract 50
-- TX A reads 100, TX B reads 100
-- TX A writes 50, TX B writes 50 → balance should be 0, it's 50!
-- This is the lost update problem

-- ❌ Vulnerable (race condition):
-- TX A: SELECT balance FROM accounts WHERE id=1; -- reads 100
-- TX B: SELECT balance FROM accounts WHERE id=1; -- reads 100
-- TX A: UPDATE accounts SET balance=50 WHERE id=1;
-- TX B: UPDATE accounts SET balance=50 WHERE id=1; -- overwrites!

-- ✅ Fix 1: Atomic UPDATE (no separate read)
UPDATE accounts
SET balance = balance - 50  -- DB does the math atomically
WHERE id = 1 AND balance >= 50;  -- check constraint

-- ✅ Fix 2: Pessimistic locking (FOR UPDATE)
START TRANSACTION;
SELECT balance FROM accounts WHERE id=1 FOR UPDATE; -- X-lock now
-- TX B's SELECT FOR UPDATE waits here
UPDATE accounts SET balance=balance-50 WHERE id=1;
COMMIT;

-- ✅ Fix 3: Optimistic locking (version column)
SELECT balance, version FROM accounts WHERE id=1; -- returns version=5
UPDATE accounts
SET balance=balance-50, version=version+1
WHERE id=1 AND version=5;  -- fails if version changed
-- Check affected_rows: if 0 → conflict → retry

SC3Scenario: Safely delete 100M old rows without downtimePro▾
-- ❌ NEVER: one shot DELETE on 100M rows
-- Locks table, creates massive undo log, kills replication
DELETE FROM logs WHERE created_at < '2023-01-01'; -- ❌

-- ✅ Method 1: Chunked delete with throttle (pt-archiver)
pt-archiver \
  --source h=localhost,D=mydb,t=logs \
  --where "created_at < '2023-01-01'" \
  --purge \
  --limit 1000 \
  --sleep 0.1 \  -- 100ms pause between chunks
  --progress 5000

-- ✅ Method 2: Application-level loop
DELIMITER //
CREATE PROCEDURE ChunkedDelete()
BEGIN
  DECLARE v_rows INT DEFAULT 1;
  WHILE v_rows > 0 DO
    DELETE FROM logs
    WHERE created_at < '2023-01-01'
    LIMIT 1000;           -- small chunks

    SET v_rows = ROW_COUNT();
    DO SLEEP(0.05);       -- 50ms breather
  END WHILE;
END//

-- ✅ Method 3 (BEST at scale): Partition DROP
-- If table is partitioned by date, drop the old partition:
ALTER TABLE logs DROP PARTITION p_2022;  -- Instant! Deletes 100M rows in ms
🔬 Lesson from experience: Every large-scale data cleanup should use partitioning + DROP PARTITION if at all possible. Plan your partition strategy at table creation time. Retrofitting partitions onto an existing 500GB table requires a full table rebuild — plan this during off-peak with gh-ost.
